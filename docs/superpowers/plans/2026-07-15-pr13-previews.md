# PR 13 — Previews: fullscreen lightbox (image / pdf.js / range-capped text)

Spec: `docs/superpowers/specs/2026-07-15-bare-bucket-v1-design.md` §7.5 ("Fullscreen lightbox: dimmed overlay, ←/→ steps through siblings in the current folder, Esc closes; actions in the overlay (download, favorite, delete). Content types: images (direct stream), PDF (pdf.js, range-requested), text (range-requested, capped preview size). Everything else: metadata + download button."), §5.2 (previews use ranged GETs). Depends on PRs 11–12 (download/delete flows the overlay actions reuse).

## Global constraints (binding)

- [B1] Presigned URLs are bearer tokens: never logged, never persisted; the lightbox holds them only in component-local state and clears them (img src removed, object URLs revoked, pdf.js doc destroyed) on close/navigate.
- [B2] Shared modal/overlay base (PR 10/12/13 carry): extract `web/src/components/ModalBase.svelte` (backdrop, Escape-to-close, `role="dialog" aria-modal`, initial-focus hook, FOCUS TRAP — Tab cycles inside) and REFIT ConflictModal + DeleteConfirmModal onto it with zero behavior change (their existing tests/behavior stay green; DeleteConfirm gains post-error re-focus per PR 12 triage). The lightbox uses the same base (fullscreen variant).
- [B3] Lightbox keyboard contract: Esc closes; ←/→ step through PREVIEWABLE-OR-NOT siblings (all files, not folders) of the current folder listing in listing order, wrapping is NOT required (stop at ends); keys are inert while a nested modal (delete confirm) is open.
- [B4] Overlay actions reuse existing flows verbatim: download button → same `downloadFile` handler (FSA/fallback); favorite → `session.toggleFavorite`; delete → DeleteConfirmModal → on success the lightbox advances to the next sibling or closes if none. No parallel implementations.
- [B5] Content-type routing by extension/content_type: images (`image/*`: png/jpg/jpeg/gif/webp/svg/avif) → `<img>` with presigned GET URL; PDF → pdf.js; text (`text/*` + md/json/log/csv/yaml/toml/xml, and application/json) → ranged GET capped at **256 KiB** with a "Preview truncated — download for the full file" notice when the object is larger; everything else → metadata panel (name, size, modified, content-type) + download button.
- [B6] Text preview uses a ranged fetch (`Range: bytes=0-262143`) on the presigned URL; render as escaped plain text (`<pre>`), never innerHTML. SVG previews render via `<img>` (never inline SVG injection) so scripts can't execute.
- [B7] PDF: `pdfjs-dist` npm dependency (spec-sanctioned), lazy-loaded (dynamic `import()`) so the main bundle doesn't grow; worker wired the Vite way (`?url` import of the worker asset). Feed pdf.js the presigned URL and let it range-request; if the provider/CORS blocks Range, pdf.js falls back to full fetch on its own — acceptable. Render pages into canvas with basic prev/next page controls + page count for multi-page docs.
- [B8] Opening a preview: clicking a FILE row (list) or tile (grid) opens the lightbox. File rows/tiles therefore become real interactive elements exactly like the folder rework in PR 12 [a11y]: real button (or equivalent) around the name/tile, Enter/Space opens, existing action buttons stay independently clickable (stopPropagation preserved).
- [B9] Loading/error states: spinner while fetching; a preview fetch failure (403 expired/404 deleted out-of-band) shows an in-overlay error with a Retry (re-presign) button — the lightbox does NOT close itself; secret/URL hygiene in the error text (status + provider message only, never the URL).
- [B10] CORS note: ranged GETs need `Range` in AllowedHeaders and `Content-Range`/`Accept-Ranges` in ExposeHeaders — validate against MinIO live; document in the CORS module docs (core/src/s3/multipart.rs CORS section) now, full per-provider docs in PR 15.
- [B11] New wasm surface: none expected — `presign_get` (PR 11) covers image/pdf/text URLs. Don't add wasm methods.
